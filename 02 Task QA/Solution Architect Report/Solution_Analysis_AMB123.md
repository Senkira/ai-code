# เอกสารวิเคราะห์ Solution: เพิ่มเงื่อนไข AMB123

**วันที่:** 26 กุมภาพันธ์ 2569
**โปรเจกต์:** OmniChannelServiceAPI
**ผู้จัดทำ:** AI Software Architect

---

## 1. Requirement สรุป

ต้องการปรับปรุงเงื่อนไขการตรวจสอบลูกค้าที่ต้องการนัดหมายยื่นเอกสาร โดยเพิ่มเงื่อนไขสำหรับลูกค้าประเภท `AMB123` ซึ่งหากลูกค้ามีสถานะนี้ ระบบจะต้อง Return HTTP Status `203` เพื่อให้ระบบ Front-end นำไปใช้งานต่อ

## 2. การวิเคราะห์ระบบปัจจุบัน (As-Is Analysis)

จากการตรวจสอบโค้ดในโปรเจกต์ `OmniChannelServiceAPI` พบว่า Logic การตรวจสอบเงื่อนไขของลูกค้าก่อนการนัดหมายนั้นอยู่ที่:

- **Service:** `OtpService.cs`
- **Method หลัก:** `AppointKYCVerify(RequestVerifyOtpModel model)`
- **Method ย่อย (หัวใจหลัก):** `GetStatuResultCode(string code)`

**Flow การทำงานปัจจุบัน:**
1.  Client ส่ง `idno` และ `otp` มาที่ endpoint `KYC/AppointKYCVerify`.
2.  `AppointKYCVerify` ทำการตรวจสอบ OTP และดึงข้อมูล `TelesaleResultCurrentCode` ของลูกค้าจาก `EagleEyeResult`.
3.  `TelesaleResultCurrentCode` ถูกส่งเข้าไปใน Method `GetStatuResultCode`.
4.  `GetStatuResultCode` ทำการตรวจสอบ `code` ที่ได้รับมาและ Return `status` กลับไป:
    - `201` สำหรับกลุ่ม Walk-in (WI).
    - `202` สำหรับกลุ่ม Direct Sales (DS).
    - `200` สำหรับกรณีอื่นๆ.

เป้าหมายของเราคือการแทรกเงื่อนไขสำหรับ `AMB123` เข้าไปใน Method `GetStatuResultCode` นี้

## 3. แนวทางการแก้ไขที่พิจารณา (Solution Options)

เราได้พิจารณาแนวทางการแก้ไขที่เป็นไปได้ทั้งหมด 3 แนวทาง ดังนี้

### แนวทางที่ 1: แก้ไขโค้ดโดยตรง (Direct Modification) - **แนวทางที่เลือก**

- **วิธีการ:** เพิ่มเงื่อนไข `else if` เข้าไปใน `GetStatuResultCode` เพื่อตรวจสอบ `code == "AMB123"` และให้ Return `status = 203`.
- **ข้อดี:** ง่าย, ตรงไปตรงมา, ความเสี่ยงต่ำ, และสอดคล้องกับ Pattern การทำงานเดิมของ Method.
- **ข้อเสีย:** มีการ Hardcode ค่า "AMB123" ในโค้ด.
- **คะแนน:** **10/10** (เหมาะสมที่สุดสำหรับ Requirement นี้)

### แนวทางที่ 2: ใช้ Configuration File ในการจัดการ (Configuration-Driven)

- **วิธีการ:** เพิ่ม Mapping ของ `AMB123` และ `203` ลงในไฟล์ `appsettings.json` แล้วแก้ไขโค้ดให้อ่านค่าจาก Config.
- **ข้อดี:** ยืดหยุ่นสูง สามารถเพิ่มเงื่อนไขใหม่ได้โดยไม่ต้องแก้โค้ด.
- **ข้อเสีย:** ซับซ้อนกว่าแนวทางแรก และอาจเป็นการออกแบบเกินความจำเป็น (Over-engineering) สำหรับเงื่อนไขเดียว.
- **คะแนน:** 8/10

### แนวทางที่ 3: ใช้ Database ในการจัดการ (Database-Driven)

- **วิธีการ:** สร้างตาราง Mapping ใน Database และให้โค้ด Query จากตารางนั้น.
- **ข้อดี:** ยืดหยุ่นสูงสุด สามารถแก้ไข Logic ได้แบบ Real-time.
- **ข้อเสีย:** ซับซ้อนมากที่สุด กระทบ Performance และเป็นการ Over-engineering อย่างชัดเจน.
- **คะแนน:** 4/10

## 4. เหตุผลในการเลือกแนวทางที่ 1

เราตัดสินใจเลือก **แนวทางที่ 1 (Direct Modification)** เนื่องจากเป็นแนวทางที่สมดุลที่สุด โดยยึดหลักการ:

- **KISS (Keep It Simple, Stupid):** เป็นแนวทางที่เรียบง่ายและเข้าใจง่ายที่สุด
- **YAGNI (You Ain't Gonna Need It):** เราไม่ควรสร้างกลไกที่ซับซ้อนเพื่อรองรับอนาคตที่ยังไม่เกิดขึ้นจริง การเพิ่ม `if` block เพียงพอต่อความต้องการปัจจุบัน
- **Low Risk & Low Impact:** การแก้ไขจำกัดอยู่ในขอบเขตเล็กๆ และไม่ส่งผลกระทบต่อส่วนอื่นๆ ของระบบ ทำให้ง่ายต่อการทดสอบและ Rollback หากเกิดปัญหา
